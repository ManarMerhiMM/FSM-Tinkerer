Exercise 3 – Applications of Finite State Machines (FSMs)

Finite State Machines are models of computation that operate with a finite set of states and react to inputs or stimuli by moving between these states. A state represents a mode of operation at a given moment, and a transition represents how the system reacts when an event occurs. Whenever we need to describe or implement behavior that depends on patterns of events, FSMs are a natural fit. They appear in digital logic, embedded systems, communication protocols, and control software.

The two examples below connect FSMs to everyday engineering systems: a traffic light controller and a vending machine controller.

Application 1 – Traffic Light Control System

1. Introduction

One of the most common examples of an FSM is a traffic light controller at an intersection. It controls the flow of vehicles and pedestrians using light signals (Green, Yellow, Red) based mainly on timing. The FSM guarantees that opposite directions are never green at the same time and that pedestrians are only allowed to cross at safe moments.

2. FSM Definition

M = (Q, Σ, δ, q₀, F)

  • Q – Set of states
  • Σ – Set of input symbols or events
  • δ – Transition function
  • q₀ – Initial state
  • F – Final or accepting states (not applicable here)

Set of States (Q)

State Description
S₁ Green Light (Main Road)
S₂ Yellow Light (Main Road)
S₃ All-Red (Safety Delay)
S₄ Green Light (Side Road)
S₅ Yellow Light (Side Road)
S₆ Pedestrian Walk
S₇ Pedestrian Flashing Don’t Walk

Input Symbols (Σ)

  • t_G_main, t_Y_main, t_G_side, t_Y_side, t_AR – timer expirations for the lights
  • t_PW, t_PF – pedestrian walk and flashing timers expire
  • t_AR_PED, t_AR_NOPED – all-red expired with / without a pending pedestrian request
  • reset – optional global reset

Output Function (λ)

State Active Outputs
S₁ (Main Green) Main_G = 1, Side_R = 1, Ped_S = 1
S₂ (Main Yellow) Main_Y = 1, Side_R = 1, Ped_S = 1
S₃ (All-Red) Main_R = 1, Side_R = 1, Ped_S = 1
S₄ (Side Green) Side_G = 1, Main_R = 1, Ped_S = 1
S₅ (Side Yellow) Side_Y = 1, Main_R = 1, Ped_S = 1
S₆ (Ped Walk) Ped_W = 1, Main_R = 1, Side_R = 1
S₇ (Ped Flash) Ped_F = 1, Main_R = 1, Side_R = 1

3. Explanation of Behavior

The FSM starts in S₁ (Main Green). After the main-road green timer expires, it moves to yellow then to an all-red safety state. Depending on whether a pedestrian has requested to cross, the FSM either switches to S₆ (Pedestrian Walk) or gives green to the side road. Eventually the controller cycles back to the main-road green. Because both the states and transitions are carefully defined, there is never a situation where both directions are green at the same time.

4. Example Run

One example sequence is: Main Road Green (30 s) → Yellow (5 s) → All-Red (2 s) → Pedestrian Walk (10 s) → Pedestrian Flash (5 s) → Side Road Green → then repeat.

5. Advantages

  • Predictable and deterministic behavior
  • Easy to extend to multiple intersections
  • Ensures safety and proper coordination of vehicles and pedestrians

Application 2 – Vending Machine Controller

1. Introduction

A vending machine that accepts coins and dispenses products based on the amount paid can also be modeled as an FSM. Each state corresponds to the current credit or operating mode, and transitions occur when coins are inserted, items are selected, or special conditions like timeout or out-of-stock are detected.

2. FSM Definition

M = (Q, Σ, δ, q₀, λ)

  • Q – Set of states
  • Σ – Set of input symbols and events
  • δ – Transition function
  • q₀ – Initial state
  • λ – Output function

Set of States (Q)

State Description
S₁ Idle
S₂ Accepting Coins
S₃ Sufficient Credit
S₄ Dispensing Product
S₅ Returning Change
S₆ Refunding
S₇ Out of Stock

Input Symbols (Σ)

  • coin_inserted – coin inserted, adds to credit
  • credit_ge_price – available credit ≥ item price
  • cancel / timeout – user cancels or waits too long
  • select_ok – valid item selected and in stock
  • select_empty – selected item is out of stock
  • done – operation finished (dispense / change / refund)
  • stock_empty – machine detects no stock
  • stock_ok – machine restocked and ready

Output Function (λ)

State Active Outputs
S₁ (Idle) Display = “Insert Coin”, Ready
S₂ (Accepting) Display = “Balance: X”
S₃ (Sufficient Credit) Display = “Select Item”
S₄ (Dispensing) Dispense, Lock
S₅ (Returning Change) Return_Ch
S₆ (Refunding) Refund, Display = “Refunding”
S₇ (Out of Stock) Display = “Out of Stock”, Lock

3. Explanation of Behavior

The vending machine starts in the Idle state waiting for a coin. When coins are inserted it moves to an accepting state and accumulates credit. Once the credit is sufficient, the user is prompted to select an item. A valid selection leads to a dispensing state, followed by returning any change. If the user cancels or the selected item is empty, the FSM moves to a refund state and returns the balance before going back to Idle. When stock is completely depleted, the machine indicates “Out of Stock” until it is restocked.

4. Example Operation Traces

  • Idle → coin_inserted → coin_inserted → Sufficient Credit → Select Item → Dispensing → Returning Change → Idle
  • Accepting → cancel → Refunding → Idle
  • Stock Empty → Out of Stock → Restocked → Idle

5. FSM Properties

  • Deterministic transitions for each possible input
  • Safe handling of errors and refunds
  • Easy to extend for multiple products and payment methods

Conclusion

These two examples show how FSMs provide a structured way to capture system behavior. In the traffic light controller, FSMs enforce safe signal ordering and coordination. In the vending machine, they manage monetary logic and user interaction. Because FSMs are deterministic and well-defined, they are a core tool in digital design and control applications.
